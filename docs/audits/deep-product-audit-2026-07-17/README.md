# 剧创 AI 深度产品审计

日期：2026-07-17  
范围：当前本地运行版本（桌面 1440×1000、移动端 390×844）、核心用户路径、当前 API 状态、代码与测试健康度。  
结论口径：截图只用于证明当前可见体验；架构结论同时参考当前代码、运行时接口和测试结果。

## 一句话结论

这已经是一套有真实产品感的“AI 短剧镜头生成与审核工作台”，但还不是一套闭环、统一、可信的“短剧生产操作系统”。桌面端项目入口、剧集总览、镜头工作台和生成任务已经形成可信骨架；真正阻碍继续迭代的不是视觉，而是两套工作流并存、阶段状态不统一、后半段生产链路为空，以及当前代码与运行时存在版本漂移。

## 当前体验最好的部分

1. **入口明确。** 项目页能用“一张近期项目主卡 + 项目库”快速恢复工作，视觉焦点和主按钮都清楚。
2. **剧集总览很接近真实工作台。** 99% 进度、下一步、场景卡、待处理、AI 建议和生成动态放在一个页面，用户能快速知道“现在该做什么”。
3. **桌面镜头工作台是核心亮点。** 场景树、中央画面、镜头条、版本区和参数检查器形成稳定的三栏生产布局；生成、动态视频、复核三种动作也有层级。
4. **版本与审核意识是对的。** 候选版本不直接覆盖当前版本；角色一致性复核、局部修改影响分析、时间线批准与导出门禁都符合生产工具的基本方向。
5. **基础可访问性考虑明显。** 有跳过链接、原生 `dialog`、可见标签、减少动画／透明度／高对比度媒体查询，状态多数同时使用文字而非只用颜色。

## 最关键的体验问题

### P0：两套生产路径并存，用户不知道哪套才是真正主流程

当前同一项目同时存在：

- 旧路径：项目 → 第 1 集 → 场景／镜头 → 完整小样；
- 新路径：项目简报 → 故事与剧本 → 前期资产 → 动态分镜 → 正式制作与交付。

旧路径已有 8 个镜头、第 16 版时间线和大量生成任务；新路径对应的动态分镜为 0、音频节点为 0、时间线轨道为 0。页面却都可以直接访问。结果是用户会同时看到“99% 就绪”“完整小样正在组装”“正式媒体仍在生成”“0 个任务运行中”。

建议：只保留一个可见的项目阶段地图。旧项目需要显式显示“旧版项目”，提供一次性迁移或只读入口；未开启的新阶段不应出现在主路径。

### P0：状态没有单一事实源

当前可见矛盾包括：

- 项目卡显示“小样就绪、99%”；
- 剧集页显示 0 个运行中任务；
- 完整小样页显示“小样正在组装”；
- 正式制作页显示“正式媒体仍在生成”；
- 后台 Worker 实际是 `IDLE`；
- 任务页里旧版 Preview Timeline 已显示“组装完成”。

这不是文案问题，而是不同页面分别从 Project、Timeline、Job、Workflow v2 数据推断状态。建议引入服务端 `ProjectReadiness` 聚合合同，统一返回当前阶段、阻断项、活动任务、下一动作和可恢复动作；所有页面只消费这一个状态。

### P0：角色一致性门禁的信任风险

S04 的自动相似度为 0%，系统同时允许点击高强调主按钮“一致并应用第 5 版”。审核中心又显示“结构一致性通过”。即使两者是不同检查，用户也无法理解其差异。

建议：0% 或 `REVIEW_REQUIRED` 时，主动作应改为“人工覆盖并应用”，要求勾选确认并填写原因；“结构一致性”“身份一致性”“连续性”应拆成清楚的三个指标，并显示各自证据。

### P0：新版后半段不是可用功能，而是可访问的空壳

当前前期资产页只有角色候选，Look、声音、场景、道具、视觉圣经均为空；动态分镜是空态；正式制作页长期显示生成中，但没有活动任务、ETA、失败原因或恢复动作。

建议：在数据合同与真实生成链路完成前，将这些页面收敛为一个“即将开放／需要升级项目”的明确状态；完成后再按 G3–G5 一次性开放。

### P1：移动端镜头工作台不适合实际操作

390px 视口下，场景树使用横向卡片，展开后的第二个场景被裁切；中央画面前出现较大的空白；底部导航长期覆盖镜头条和后续控件。移动端可以做审核，但不应完整复制桌面三栏生产台。

建议：移动端切换为“画面 → 待审动作 → 参数抽屉”的单任务模式；场景树改为顶部下拉或底部抽屉；底部导航在全屏镜头工作台隐藏。

### P1：长页面和空页面同时存在

项目简报把定位、市场、语言、平台、约束、版本信息和已批准方案放在一个超长页面；审核中心却只有一张卡和大量空白；任务中心有 18 条任务但缺少搜索、任务类型、模型、费用、错误详情与排序。

建议：

- 简报拆为“故事核心 / 受众与市场 / 生产规格 / 约束”四个折叠段；
- 审核中心增加范围、类型、风险、状态筛选，并支持批量通过／退回；
- 任务中心增加任务详情抽屉、模型与请求 ID、耗时、积分、失败链路和恢复建议。

## 欠缺的关键功能

按优先级排序：

1. **统一工作流与旧项目迁移。** 这是继续增加功能前的前置条件。
2. **项目就绪度与卡点诊断。** 明确当前阶段、为什么卡住、正在跑什么、下一步是什么。
3. **完整角色资产包。** 正脸、3/4 侧脸、侧脸、全身、服装／Look 版本、标志物、负面约束和身份档案；不能用“手部画面”充当角色参考。
4. **真正的动态分镜与正式媒体链路。** 镜头视频、TTS、BGM、环境音、SFX、口型降级、多轨时间线、整片 QC 和交付矩阵要在一个真实项目中走通。
5. **生产级审核中心。** 对比、QC 证据、提示词、模型、成本、批注、退回原因、审核历史和批量操作。
6. **批量生成与预算控制。** 按场景／镜头批次生成，设置并发、模型、积分上限、失败策略和自动暂停条件。
7. **任务可观测与恢复。** 卡死检测、租约过期解释、重试范围、从失败节点续跑、活动任务与页面状态自动对账。
8. **导出前的真实权利与内容安全检查。** 角色授权、声音授权、音乐授权、临时资产、水印、内容风险与平台规格分开呈现。

团队协作、多租户和自动发布可以继续保持非目标；它们不是当前 MVP 的首要缺口。

## 信息架构判断

全局层面“项目 / 生成任务 / 审核中心 / 设置”是合理的。问题集中在项目内部：

- 项目阶段没有持久化导航，只能从页面按钮或直接路径进入；
- 面包屑是文字而非可点击路径；
- “角色形象”和“前期资产”重复；
- “完整小样”和“正式制作与交付”重复；
- “审核中心”和镜头内审核重复；
- 当前项目入口会跳到剧集页，而项目总览又承担简报编辑。

建议的信息架构：

`项目总览 → 1 简报 → 2 故事/剧本 → 3 角色与美术资产 → 4 分镜/镜头生产 → 5 成片/交付`

“生成任务”和“审核中心”保留为跨阶段横向工具；不再作为另一套生产路径。

## 架构关键问题

### 1. 运行时与仓库版本漂移

仓库迁移头为 `0023_character_visual_identity`，当前正在运行的后端仍报告当前与预期均为 `0022_narrative_targeting`。这说明当前 UI 连接的是未重启的旧后端进程。审计截图代表真实运行体验，但不完全代表仓库最新后端能力。

### 2. 功能开关没有形成真正边界

运行时返回 Workflow v2 相关开关均为关闭；代码中这些开关主要被暴露给配置页，未统一控制路由、API 或页面可见性。因此用户仍能访问空的新流程。

### 3. 双工作流导致双数据模型、双页面和双状态机

旧版 Project/Episode/Scene/Shot/Timeline 与新版 Story/Preproduction/Storyboard/Audio/Multitrack 同时存在，且没有明确迁移合同。这是当前状态矛盾与 IA 分裂的根因。

### 4. 前端刷新模型重复

全局 StudioContext 同时使用 SSE 和 3 秒轮询；任务页、故事页、角色页、分镜页、制作页等又各自轮询。它能保证本地 MVP“最终刷新”，但会产生重复请求、不同快照和短暂状态回退。应统一为一个项目事件缓存与查询层。

### 5. Web 进程内单 Worker + SQLite 只适合当前本地 MVP

这个选择本身对单用户本地试点是合理的，并不是当前错误。问题是进入多项目并发、长文本生成、视频轮询或多实例部署后，Web 生命周期、Worker 生命周期和 SQLite 写竞争会绑在一起。扩展前需要独立 Worker 与 PostgreSQL；在当前 MVP 范围内可以暂不改。

### 6. 当前后端测试不是绿色

本轮结果：前端类型检查通过，13 个前端测试文件 / 56 个测试通过；后端 119 通过、7 失败。失败包括项目全图删除的外键约束、迁移恢复、Worker 注册合同、新增角色视觉流程后的关系图合同漂移等。部分是测试未同步新能力，部分是真实删除与迁移风险，不能作为可发布状态忽略。

## UX 与可访问性隐患

- 大量禁用按钮没有就地解释，用户只能猜测前置条件。
- 通知红点只要存在“最近任务”就出现，不等于未读通知，容易永久失真。
- 小号灰色元数据、浅蓝禁用按钮和半透明模态层存在对比度风险，需要实测 WCAG 对比值。
- 横向滚动的场景卡没有明显“可横滑”提示。
- 移动端固定底部导航覆盖工作台内容与关键操作区。
- 正向点：原生对话框、键盘 Esc、跳过链接、减少动画、减少透明度和增强对比度样式都已存在。

截图不能证明完整键盘顺序、屏幕阅读器播报和实际对比度合规，仍需使用真实辅助技术与自动化工具复核。

## 推荐实施顺序

1. 冻结新增页面和新 Provider，先统一 ProjectReadiness、阶段状态与旧项目迁移。
2. 用一个真实项目打通 G3→G5，所有阶段都必须有活动任务、失败态、恢复动作和真实产物。
3. 重做审核中心与任务详情，让每个生成结果都有来源、成本、证据和决定。
4. 将移动端明确定位为“查看与审核”，重构镜头工作台响应式结构。
5. 修复后端 7 个失败测试并建立“迁移头 = 运行时预期 = 数据库当前”的启动门禁。
